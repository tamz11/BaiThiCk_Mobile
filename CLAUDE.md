# CLAUDE.md

## Project Summary
This project is a Flutter mobile app for booking medical appointments. Users can register/login, browse doctors, view doctor profiles, book appointments, and manage appointment history.

The app uses Firebase services for authentication and data storage. In the current implementation, doctor data is read from Firebase Realtime Database, while user profile, diseases, and appointments are handled in Cloud Firestore.

## Run, Test, Format, Lint

### Install dependencies
```bash
flutter pub get
```

### Run app
```bash
flutter run -d chrome
```

Optional targets:
```bash
flutter run -d macos
flutter run -d ios
flutter run -d android
```

### Tests
Run all tests:
```bash
flutter test
```

Run a single test file:
```bash
flutter test test/widget_test.dart
```

### Format code
Format entire project:
```bash
dart format .
```

Format a folder:
```bash
dart format lib test
```

### Lint / static analysis
```bash
flutter analyze
```

## Key File Locations

### App entry and routing
- lib/main.dart: App bootstrap, Firebase init, named routes.
- lib/mainPage.dart: Bottom navigation and main tab pages.

### Firebase configuration
- lib/firebase_options.dart: Generated by FlutterFire CLI. Do not edit manually.
- firebase.json: FlutterFire platform mapping.
- android/app/google-services.json: Android Firebase config.
- ios/Runner/GoogleService-Info.plist: iOS Firebase config.
- macos/Runner/GoogleService-Info.plist: macOS Firebase config.

### Data layer
- lib/data/realtime_doctors_repository.dart: Realtime Database source for doctors.
- lib/data/mock_doctors.dart: Local mock data fallback utilities.

### Firestore UI/data access
- lib/screens/bookingScreen.dart: Creates appointments under appointments/{uid}/pending and appointments/{uid}/all.
- lib/firestore-data/myAppointmentList.dart: Reads pending appointments.
- lib/firestore-data/appointmentHistoryList.dart: Reads appointment history.
- lib/screens/userProfile.dart and lib/updateUserDetails.dart: Reads/writes users/{uid} profile fields.
- lib/screens/disease.dart: Reads diseases collection.

### Models
- lib/model/bannerModel.dart
- lib/model/cardModel.dart

### Tests
- test/widget_test.dart (currently placeholder test)

### Project config
- pubspec.yaml: Dependencies, assets, SDK constraints.
- analysis_options.yaml: Lint/analyzer rules.

## Conventions Used
- State is mostly managed in StatefulWidget screens and simple controllers.
- App routing currently uses named routes in MaterialApp.
- Firebase is initialized via DefaultFirebaseOptions.currentPlatform.
- Naming is mixed legacy style (example: bookingScreen.dart and booking_screen.dart). Follow existing naming in touched module, avoid broad renames.
- For doctor data, repository currently expects Realtime Database URL in lib/data/realtime_doctors_repository.dart.

## Safe Commands
Use these commands freely during development:

```bash
flutter pub get
flutter analyze
flutter test
dart format lib test
flutter run -d chrome
flutter run -d macos
```

For Firebase setup (manual/devops step):

```bash
firebase login
flutterfire configure
```

## Commands To Avoid
Avoid these unless explicitly needed and approved:

- git reset --hard
- git checkout -- .
- force-deleting platform folders
- manually editing generated Firebase files without rerunning FlutterFire
- flutter pub upgrade --major-versions on shared branches without coordination

## Developer Workflow Rules
1. Keep Firebase project configuration consistent across all files (firebase_options.dart, google-services.json, plist files, and realtime DB URL).
2. If doctors data is missing, first verify Realtime Database URL and seeded doctor JSON.
3. Do not manually rewrite generated files under FlutterFire outputs; rerun flutterfire configure instead.
4. Before pushing changes, run: dart format lib test, flutter analyze, flutter test.
5. Keep changes scoped. Avoid unrelated refactors in the same commit.
6. If adding new features (calendar, chatbot, notifications), document setup and required secrets in README and this file.
